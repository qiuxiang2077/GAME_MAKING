# 外部协作指南：美术与音乐资源管理

## 概述
本文档说明核心开发团队（2人：程序/策划）如何与外部美术和音乐同学协作，确保资源生产与开发进度协调一致。

## 团队结构澄清

### 核心团队（2人）
- **角色**：程序开发 + 游戏策划
- **职责**：
  - Godot引擎开发与编程
  - 游戏系统设计
  - 关卡策划与数值平衡
  - 项目管理与进度跟踪
  - 美术/音乐需求定义
- **时间投入**：每人每周约15小时，总计30小时/周

### 外部协作同学
1. **美术同学**
   - **角色**：美术资源生产者
   - **协作模式**：按需求生产像素艺术资源
   - **交付内容**：角色精灵、武器道具、UI元素、场景贴图、动画
   - **工具**：Aseprite / Pixelorama / Photoshop

2. **音乐同学**
   - **角色**：音频资源生产者
   - **协作模式**：按需求生产音频资源
   - **交付内容**：背景音乐、音效、环境音频
   - **工具**：FL Studio / LMMS / 其他DAW软件

## 协作流程

### 总体流程
```
核心团队分析需求 → 填写需求文档 → 讨论确认细节 → 外部同学生产 → 验收集成 → 反馈迭代
```

### 详细步骤

#### 1. 需求分析与规划
- **时间**：每周开发计划会议时同步进行
- **内容**：
  - 确定下周需要的美术/音乐资源
  - 评估资源优先级和依赖关系
  - 预估外部同学所需时间

#### 2. 需求文档填写
- 使用标准化模板：[ART_REQUIREMENT_TEMPLATE.md](ART_REQUIREMENT_TEMPLATE.md) 或 [AUDIO_REQUIREMENT_TEMPLATE.md](AUDIO_REQUIREMENT_TEMPLATE.md)
- 填写要点：
  - 清晰描述使用场景和功能需求
  - 提供具体视觉/音频参考
  - 明确技术规格和交付要求
  - 设置合理的期望完成日期

#### 3. 需求确认会议
- **频率**：每个需求开始时一次简短会议（15-30分钟）
- **参与者**：核心团队 + 相关外部同学
- **议程**：
  - 解读需求文档内容
  - 确认技术细节和风格要求
  - 讨论时间安排和交付节点
  - 解答外部同学的疑问

#### 4. 生产与沟通
- **沟通频率**：
  - 美术需求：每3-4天一次进度更新
  - 音乐需求：提供初步版本试听反馈
- **沟通工具**：
  - 微信：日常快速沟通
  - 飞书文档：正式需求和反馈记录
  - 文件共享：Google Drive / 网盘

#### 5. 验收与集成
- **验收标准**：
  - 符合需求文档中的技术规格
  - 风格与现有资源统一
  - 功能实现符合游戏需求
- **验收流程**：
  1. 外部同学提交完成版本
  2. 核心团队测试集成效果
  3. 提供具体、可操作的反馈
  4. 确认验收或提出修改建议

#### 6. 反馈与迭代
- **反馈原则**：
  - 具体而非模糊（"颜色太暗" vs "需要调亮20%")
  - 功能导向而非主观偏好
  - 尊重专业创作，给予合理创作空间
- **迭代限制**：通常不超过2轮修改，避免无限返工

## 时间管理建议

### 需求提交流程优化
1. **提前规划**：核心团队需提前1-2周规划资源需求
2. **批次处理**：相似资源批量提交，减少沟通成本
3. **优先级明确**：标记高优先级资源，确保关键路径不受阻

### 缓冲时间设置
- **美术资源**：简单资源（图标、道具）预留3-5天，复杂资源（角色动画）预留1-2周
- **音乐资源**：音效预留2-3天，背景音乐预留1-2周
- **紧急需求**：预留额外20%缓冲时间

### 进度同步机制
1. **每周简要同步**：核心团队周会时同步外部资源状态
2. **关键节点检查**：复杂资源生产中期检查进度
3. **风险预警**：遇到延期风险尽早沟通调整计划

## 常见场景处理

### 场景1：资源未按时完成
**处理流程**：
1. 核心团队使用占位资源继续开发
2. 与外部同学沟通延期原因和新的时间节点
3. 调整后续资源需求优先级
4. 记录延期原因，优化未来时间预估

### 场景2：资源风格不一致
**处理流程**：
1. 提供更具体的风格参考和规范
2. 建立项目风格指南文档
3. 对已完成的资源进行统一调整
4. 后续需求中加强风格描述

### 场景3：功能需求变更
**处理流程**：
1. 评估变更的必要性和影响范围
2. 更新需求文档并记录变更原因
3. 与外部同学沟通变更内容和时间影响
4. 调整项目计划和时间安排

### 场景4：质量不符合预期
**处理流程**：
1. 提供具体的、可操作的改进建议
2. 讨论技术实现难点和解决方案
3. 考虑简化需求或调整验收标准
4. 如必要，寻找替代解决方案

## 文档与版本管理

### 需求文档管理
1. **命名规范**：
   - 美术：`ART_需求描述_日期.md`
   - 音乐：`AUDIO_需求描述_日期.md`
2. **版本控制**：每次重大修改保存新版本
3. **状态跟踪**：使用需求文档中的状态字段跟踪进度

### 资源文件管理
1. **目录结构**：
   ```
   /Assets/Sprites/          # 美术资源
   /Assets/Audio/           # 音频资源
   /Assets/Placeholders/    # 占位资源
   ```
2. **命名规范**：遵循项目统一的命名规则
3. **版本备份**：重要版本资源单独备份

## 成功协作要点

### 对核心团队的要求
1. **需求清晰化**：能准确描述需要什么，为什么需要
2. **专业尊重**：尊重美术/音乐同学的专业判断
3. **及时反馈**：尽快提供验收反馈，避免阻塞
4. **计划性**：提前规划资源需求，避免紧急需求

### 对外部同学的要求
1. **进度透明**：及时沟通进展和遇到的问题
2. **规范遵循**：遵循项目技术规范和风格要求
3. **主动沟通**：遇到理解不明确时主动询问
4. **质量保证**：确保交付资源符合验收标准

### 沟通最佳实践
1. **书面记录**：重要决定和需求通过文档确认
2. **定期同步**：保持适度的沟通频率
3. **正向反馈**：对好的工作给予肯定和感谢
4. **问题导向**：遇到问题时共同寻找解决方案
